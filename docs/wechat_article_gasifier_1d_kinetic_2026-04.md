# 一个一维气流床煤气化动力学模型，做到什么程度才算“可用”？

这篇文章介绍另一个正在进行中的项目：

`gasifier-1d-kinetic`

它是一个面向气流床煤气化炉的 **1D 轴向动力学建模工具**。

如果说 `gasifier-model` 更关心“热力学允许系统走到哪里”，那么这个项目更关心的是：

在给定停留时间、反应速率、热损分布和轴向推进路径的前提下，系统实际上能走到哪里。

写这篇文章的时机，是这个项目刚刚完成一轮比较关键的整理。到目前为止，它已经不再只是一个“能算几条文献案例”的研究脚本，而开始具备比较清晰的工程形态：

- 求解器接口已经收口到 `minimize`、`newton_fd` 和 `jax_jit`
- `jax_jit` 路线已经不再只是实验分支，而是被明确定位为在线热态服务的目标路径
- 当前正式验证数据源已经统一到 `data/validation_cases_final.json`
- 这份正式数据源按 `Pilot / Industrial`、`Dry-fed / Slurry-fed` 分层，合计 35 个 canonical cases
- UI、回归测试、预编译脚本和服务化说明已经开始形成一条完整链路

这还不是最终状态，但已经到了值得系统记录一次的节点。

我之前写过一篇平衡模型的文章。那篇文章讲的是 `gasifier-model`。这篇文章算是它的“镜像篇”。两个项目放在一起看，会更容易理解：

- 平衡模型擅长回答什么问题
- 动力学模型擅长回答什么问题
- 两者为什么都重要，但不能互相替代

## 一、这个模型能做什么，不能做什么

`gasifier-1d-kinetic` 的核心定位，是一个 **轴向推进的一维动力学模型**。

它不是直接假设出口已经达到平衡，而是把气化炉沿长度方向切成一系列 cell，逐格求解：

- 气相与固相的质量守恒
- 全局能量平衡
- 异相反应速率
- 均相反应速率
- WGS 与燃烧区判据

给定煤质、进料、压力、氧煤比、汽煤比、热损和几何尺度之后，它试图回答的问题是：

如果煤粒和气相真的按照这套动力学与能量路径沿着炉长推进，出口温度和合成气组成会走到哪里？

这类模型的优势也很明确：

- 能表达“起燃区、放热峰、后续降温”这类轴向现象
- 能把停留时间、网格、热损分配和动力学速率显式纳入
- 更适合解释为什么某些工况即使热力学允许，也未必真的走得到
- 更适合做“机理诊断”而不只是做一个出口点的拟合

但它的局限也同样明确：

- 模型复杂度明显高于平衡模型
- 收敛、初值、网格和数值稳定性会真正影响结果
- 某些工业偏差仍然会混合了结构误差、经验闭合和数据口径问题
- 你不能指望它因为“更复杂”就自动变成万能预测器

所以，这个项目真正重要的工作，并不只是把反应写得更多，而是逐步把这件事说清楚：

这个模型到底在回答什么问题，它的边界又在哪里。

## 二、为什么这个项目的重点不是“再多加几个反应”，而是把求解路线收口

这轮整理里，最重要的一件事，不是又往模型里补了多少新机理，而是把公开求解路线收口到了三条：

- `minimize`
- `newton_fd`
- `jax_jit`

这件事看起来像代码结构整理，但本质上是在重新定义项目的主线。

以前这类数值模型很容易出现一个问题：历史上为了调试、试验和赶进度，会堆出很多求解入口。短期看很方便，长期看却会带来两个后果：

- 用户不知道哪个才是“应该用的”路径
- 回归验证会越来越难，因为每条路线都有自己的口径漂移

现在这三条路线的分工被明确了：

- `minimize` 负责稳，是历史基线
- `newton_fd` 负责在 NumPy / SciPy 主线上做速度折中
- `jax_jit` 负责在稳定 shape 上追求热态 wall time 和吞吐

这一步很重要，因为只有把路线收口，后面的验证、服务化和 UI 才有可能真正一致。

## 三、三种 Solver：为什么它们都保留着，但真正的重点已经转向 JAX

### 1. `minimize`

`minimize` 本质上是 SciPy `least_squares(method="trf")` 路线。

它的优点很简单：

- 稳
- 失败时更容易诊断
- 适合作为所有新路线的 reference

它的缺点也同样简单：

- 慢
- 多 cell 累积后 wall time 明显

所以它更像基线，而不是最终在线路径。

### 2. `newton_fd`

`newton_fd` 是阻尼 Newton + 有限差分 Jacobian。

它比 `minimize` 更像“工程折中版”：

- 比纯 SciPy 基线更快
- 仍然留在 Python / NumPy 语境里，调试成本不高
- 但对初值和 Jacobian 更敏感

如果把整个项目理解成一个正在逐步走向工程可用的系统，那么 `newton_fd` 更像中间桥梁：它不是终局，但它非常有用。

### 3. `jax_jit`

`jax_jit` 才是这个项目这一轮最值得记录的变化。

但如果只把它理解成“把 Python 代码翻译成 JAX，然后跑得更快”，其实还是说浅了。

它真正重要的原因有两层：

- 第一层，是它更适合承接这个模型里最难求的那部分刚性方程
- 第二层，是它在完成编译后，确实能把重复求解的 wall time 压下来

先说第一层。

这个 1D 动力学模型里，最难求的一格通常不是后面的温和区，而是 `cell0`。

因为 `cell0` 经常同时承担几件事：

- 点火
- 氧气快速消耗
- 温度陡升
- 异相/均相反应源项突然放大
- 后续各格的初值和路径分支选择

这会让 `cell0` 形成非常典型的刚性特征：

- 状态变量之间耦合很强
- 一点点温度变化就可能显著改变速率
- 数值上很容易出现“这一小步没走对，后面整炉都偏掉”的情况

所以这个项目里真正困难的地方，从来都不只是“全炉一共多少个 cell”，而是：

**第一格起燃时那组高度非线性、强耦合、带刚性的方程，到底能不能稳定而一致地解出来。**

从这个角度看，JAX 的价值不只是快，而是更适合把这种数值路径收进一套可编译、可重复、可一致执行的计算图里。

再说第二层，也就是速度。

这里也要把一句话说得非常精确：

`jax_jit` 的优势是 **热态优势**，不是 **冷启动优势**。

它快，不是因为 “JAX 天然神奇”，而是因为：

- Python 路线本质上还是解释执行
- 循环、函数调用、数组操作和求导近似都要反复经过 host 侧调度
- JIT 路线则会把一整段数值计算编译成更紧凑的执行形式

换句话说，`jax_jit` 的速度优势来自：

- 先把计算图编译好
- 再用编译后的执行路径反复跑同类问题

这也是为什么它在第一次运行时不一定快，但在同 shape、同类型请求的重复计算里，会越来越有优势。

仓库里现有单 case 实测已经很能说明问题：

- `minimize`: `1.3035s`
- `jax_jit` 第一次（含 compile）: `9.5113s`
- `jax_jit` 第二次（热态）: `0.6205s`

也就是说，冷启动时它可能更慢；但一旦进入热态，在重复求解场景下，它就开始体现出价值。

所以现在项目里关于 `jax_jit` 的正确表述，不应该只是：

“另一个更快的 solver”

而应该是：

**一个更适合承接刚性点火方程、并且在编译完成后适合在线热态服务的目标执行路径。**

## 四、为什么 `jax_jit` 值得认真做，而不是“先放着以后再说”

很多 JAX 路线的项目，最后都会卡在一个尴尬状态：

- 代码能跑
- benchmark 也能出几张图
- 但因为首跑太慢、调试太难、部署太麻烦，最后在线上根本没人敢用

这个项目这轮比较关键的一点，是开始正面解决这个问题，而不是绕开它。

现在仓库里已经有一套相对明确的在线化思路：

- 用 `scripts/precompile_jax_solver.py` 对常见 case 和 shape 预编译
- 把 readiness probe 建立在 precompile 完成之上
- 让在线请求直接走热态 `jax_jit`
- 把 compile 成本尽量从用户请求路径里移出去

这件事背后的工程逻辑其实很朴素：

如果一个 solver 的优势只在热态存在，那线上系统就应该尽量只把“热态版本”暴露给用户。

这也是为什么我会认为，这个项目里 `jax_jit` 的意义已经不只是“JAX 重构过了”，而是：

项目第一次开始真正具备了“把数值模型当成在线计算服务来设计”的意识。

再往前说一步，这条路线还有一个常被忽略的意义：

如果未来要继续增强 `cell0` 这类刚性区的数值处理，例如更一致的 Jacobian 口径、更稳定的点火支路选择、更系统的热态批量求解，那么“先把主路径编译化”本身就是必要前提之一。

再进一步说，`jax_jit` 的价值也不只是让一个 benchmark 表更好看。

一旦动力学模型的 **稳定性** 和 **热态速度** 同时上来，它的应用边界就会明显扩大。

以前很多机理模型虽然理论上很有价值，但在线上系统里常常会卡在几个现实问题上：

- 太慢，无法承接高频重复请求
- 太脆，遇到刚性工况容易数值失稳
- 太依赖人工调试，难以长期挂在线上

而 `jax_jit` 路线真正让人感兴趣的地方，是它开始让这个 1D 动力学模型有机会脱离“研究脚本”语境，进入更典型的工业在线应用语境。

例如，它会更适合承接以下几类工作：

- **开发 soft sensor**
  - 当现场不能直接测量炉内关键状态时，可以用机理模型结合可测信号去反演或补充难测变量
  - 如果模型本身足够稳定、热态求解足够快，它就更有可能作为软测量核心而不是离线分析工具

- **作为 digital twin 的 calculation engine**
  - 数字孪生不只是一个可视化壳子，背后需要一个能持续接收新边界条件并快速重算的 engine
  - 如果每次都要承担 Python 解释执行和冷启动开销，这件事很难做得流畅
  - 但如果 `jax_jit` 已经预热，在线重复调用的成本就会明显下降

- **作为 advanced control / real-time optimization 的 mechanism model**
  - 高级控制和实时优化最怕模型“慢且不稳”
  - 因为一旦进入闭环，模型不只是要算对，还要在可接受时间内稳定返回
  - 对这类场景来说，一个能够更好承接刚性区、并且热态 wall time 更短的动力学模型，价值会远高于“离线单次拟合更漂亮”的模型

换句话说，`jax_jit` 的优势并不只是让本地 benchmark 从几秒变成更短几秒。

它真正可能改变的是：

这个动力学模型能不能从“分析工具”进一步变成“在线计算引擎”。

## 五、验证数据源为什么要统一，为什么不能再把所有 JSON 混在一起看

这轮工作里，另一个很重要但很容易被低估的变化，是验证数据源的边界被重新划清了。

现在正式口径下的验证数据源是：

- `data/validation_cases_final.json`

它不是一个平铺的字典，而是带层次的正式数据集：

- `Pilot`
- `Industrial`
- 每层再分 `Dry-fed` / `Slurry-fed`

按当前文件内容统计，总数是 35 个 canonical cases。

这件事的价值，不只是“case 变多了”，而是项目终于开始明确：

- 哪些数据属于正式 benchmark
- 哪些旧 JSON 只是开发期或测试期遗留
- 哪些 case 名虽然相似，但物理口径其实并不相同

这一步非常重要。

因为如果验证集边界不清楚，你很容易在文档里写出一个很漂亮的平均误差，但那个平均误差其实混合了：

- 文献大写 canonical case
- pilot 小写 duplicate case
- 早期 dev/test 数据
- 甚至并不相同的 feed 口径

那种“好看但不诚实”的统计，对工程没有什么价值。

## 六、精度问题现在应该怎么看

这个项目当前更有价值的进展，不是简单宣称“JAX 版也能跑”，而是：

在正式 benchmark 口径下，`jax_jit` 已经越来越接近“与 `minimize` 同级别精度的等价实现”。

基于现有 canonical benchmark 解读，仓库里的结论大致是：

- 平均出口温度差 `|ΔT_out|` 约 `0.523 K`
- 中位出口温度差 `|ΔT_out|` 约 `0.119 K`
- 最大出口温度差 `|ΔT_out|` 约 `2.402 K`

这组数字本身并不意味着“模型已经完美”，但它至少说明了两件事：

- `jax_jit` 现在已经不是一条明显跑偏的旁路
- 它有机会在不牺牲主线精度的前提下，承接更强的数值执行路径

它更像是：

在结果质量上基本贴住主线、在执行方式上更适合热态服务化的另一种实现。

这对项目非常关键，因为如果精度和执行效率不能同时成立，在线化就没有意义。

## 七、UI 不再只是开发者面板

这个项目现在也已经有自己的 Streamlit UI。

但这轮 UI 的价值，不在于“把更多参数露出来”，而在于逐步把它从开发者调参页面，整理成一个更像工作台的东西。

现在这套 UI 已经开始尝试回答几个真正影响使用体验的问题：

- 哪个 solver 当前更适合这个场景
- `jax_jit` 的时间优势到底是冷启动还是热态
- 当前结果和 `minimize` 基线相比偏了多少
- 什么时候是在做本地诊断，什么时候是在谈线上热态服务

这听起来像界面文案问题，但本质上还是模型边界问题。

一个健康的建模工具，不只是“把参数都给你”，而是应该尽量减少用户误解：

- 不把首跑时间误当成线上服务时间
- 不把拟合能力误当成预测能力
- 不把调试视图误当成生产语义

如果说前一篇平衡模型文章里，UI 重构的主题是把 `Prediction / Calibration` 讲清楚，那么这个项目 UI 重构的主题更像是把：

- `baseline`
- `工程折中`
- `热态服务`

这三件事讲清楚。

而在这三者里，`jax_jit` 最需要被正确解释的一点就是：

它不是“本地点一下就一定更快”的按钮，而是“完成编译之后，适合稳定承接刚性动力学求解”的服务路径。

## 八、作为 Chem Portal 的一个计算模块，这个项目现在已经开始有“服务形态”

从定位上看，`gasifier-1d-kinetic` 已经不只是本地研究仓库。

它现在更像 `chem_portal` 体系下的一个计算模块，而且这个模块和 `gasifier-model` 并列：

- 一个是平衡模型
- 一个是动力学模型

这两个模块最有意思的地方，不是互相替代，而是互相补位。

平衡模型更适合做：

- 快速上限估算
- 校准工作台
- 参数扫描

动力学模型更适合做：

- 轴向剖面解释
- 反应区诊断
- 停留时间和动力学效应分析
- 热态高吞吐在线求解
- soft sensor 的机理内核
- digital twin 的 calculation engine
- advanced control / real-time optimization 的机制模型底座

而一旦你把它们都放进同一个门户里，问题就不再只是“哪个更准”，而变成：

用户现在面对的到底是哪一类问题，应该把请求路由到哪一类模型。

我觉得这是这个项目很有意思的一点。它开始让“模型选择”本身，也变成系统设计的一部分。

## 九、这个项目现在处于什么阶段

如果要用一句话概括当前状态，我会说：

它已经从“一个能算的 1D 动力学脚本”，走到了“一个开始具备验证、UI 和在线化意识，并且开始认真建设 JAX 主路径的动力学建模工作台”。

但这当然不意味着问题已经解决完了。

当前最重要的几个事实是：

- 正式验证数据源边界已经比以前清楚
- 三条 solver 的角色已经基本明确
- `jax_jit` 的价值已经从“能不能跑”转向“能否稳定承接刚性区求解，并作为热态服务使用”
- 一旦稳定性和热态速度继续提升，它就有机会进入 soft sensor、digital twin 和 advanced control 这类在线场景
- UI 正在从调参表单转向更可解释的工作台
- 但模型在部分工业工况上的结构性偏差、数值稳定性和机理口径，仍然有继续打磨空间

如果接着往下走，我觉得后面的重点大概率不会是简单地“继续堆更多机制”，而是更有针对性地回答几个问题：

- 哪些偏差主要来自动力学参数
- 哪些偏差主要来自能量闭合和热损口径
- 哪些偏差已经接近一维模型本身的结构上限
- 哪些能力应该继续放在本地诊断层，哪些能力应该真正进入在线服务层

这也是为什么我觉得，把它和 `gasifier-model` 放在一起看特别有意思。

平衡模型回答的是：

“热力学允许系统大致走到哪里？”

动力学模型回答的是：

“在给定路径和停留时间下，系统实际能走到哪里？”

这两个问题都重要，但它们不是同一个问题。

## 十、最后

做过程工业建模时，经常会有一种冲动：

只要误差还没压到满意，就继续往里加更多机理、更多参数、更多补丁。

但工程上更难的一件事，往往不是“再多做一点”，而是：

知道什么时候该承认模型的边界，然后把它整理成一个用户能正确使用的工具。

我觉得 `gasifier-1d-kinetic` 现在最值得记录的进展，不只是它多了一条 `jax_jit` 路线，也不只是它又补了多少验证案例，而是它开始更清楚地回答：

- 它是什么
- 它不是什么
- 它现在真正适合拿来做什么

尤其是，它开始更明确地回答：

- 为什么 `cell0` 这种点火高温区会成为刚性求解核心
- 为什么 JAX 不只是“为了快”，也是为了把这类路径更稳定地收进编译执行
- 为什么真正该比较的是热态服务时间，而不是第一次点击按钮的时间
- 为什么一旦稳定性和速度都过线，动力学模型就能从离线分析走向在线 calculation engine

这比再把某个误差指标单独压低一点，更重要。
